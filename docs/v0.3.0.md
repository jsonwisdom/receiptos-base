# ReceiptOS v0.3.0 — Public Verification Surface

**Codename:** `PUBLIC_VERIFIER_SURFACE_V1`  
**Status:** Alpha planning / implementation seed  
**Base release:** `receiptos-v0.2.0`

## Goal

ReceiptOS v0.2.0 made the provenance engine reproducible, anchored, replayable, and monitored. v0.3.0 makes that verification surface legible to humans.

The target is simple: an auditor, collector, or downstream builder should understand the current integrity state in under ten seconds.

## Scope

- Static public status dashboard at `docs/status.html`.
- Manifest visualizer for `evidence/anchors.json`.
- Human-readable summary of release status, authority state, known variance, and anchor linkage.
- No new cryptographic primitives.
- No dynamic backend.

## Public Verifier Surface V1

The first surface must display:

- Release: `receiptos-v0.2.0`
- Status: `VERIFIED`
- Authority: `false`
- Known variance: `revocable=true`
- Anchor manifest: `evidence/anchors.json`
- Anchor: `global-forest`
- Chain: `Base`
- EAS UID: `0xa29eab49ab595b5d93dcb07e612dc0d284c83f2038f5a226c91d7cf9c2b96aa4`
- Receipt hash: `0x0b3b7495babd70d7204bfbed9490d89f3c5d8439d1a1e521bda1891dcc7f8e33`
- Forest root: `0x9f77784722956c27e274a68cf97d4e5a5192514d61a5d7c478d56bfaccad7a61`

## Manifest Visualizer

The dashboard prioritizes the manifest visualizer above decorative UI.

The visualizer should:

1. Render the current anchor registry from `evidence/anchors.json`.
2. Show the local receipt linkage.
3. Show the EAS UID and chain metadata.
4. Make `revocable=true` visible as a known variance, not a hidden caveat.
5. Keep `authority=false` prominent.

## Acceptance Criteria

- `docs/status.html` works as a single static file.
- The dashboard is readable without running a terminal.
- The page does not claim authority.
- The page clearly distinguishes verified state from revocability variance.
- The dashboard points users back to replay commands for machine verification.

## Verification Command

```bash
git checkout receiptos-v0.2.0
bash scripts/verify-all-anchors.sh evidence/anchors.json
```

If `BASE_RPC_URL` is configured, the on-chain witness and revocation state are checked through the existing verification path.

## Out of Scope

- Multi-chain expansion.
- Dashboard backend.
- API service.
- Automated attestation renewal.
- New anchor primitives.
