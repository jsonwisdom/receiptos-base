# ROS-0006: Authorized Identity Invariant

## Status

ACTIVE

## Definition

ReceiptOS binds identity to the **Authorized Identity** of the operator.

The Authorized Identity MAY be:

- an Externally Owned Account (EOA)
- a smart contract account

## Canonical Verification

### EOA

- EIP-191 `personal_sign`
- Recovery via `recoverAddress(hashMessage(message), signature)`

### Contract Account

- ERC-1271 `isValidSignature(hash, signature)`
- MUST return magic value `0x1626ba7e`
- The contract address is treated as the Authorized Identity

## Invariant

Both mechanisms are canonical for their respective identity types.

ReceiptOS binds to the Authorized Identity rather than requiring a specific signing primitive.

Implementations MUST NOT emulate EOA behavior for contract accounts or contract behavior for EOAs.

Unknown identity types MUST fail closed.

## Replay Requirements

Every receipt MUST record:

- `verification_method`
- `signer_address`
- `message_hash`
- `signature`
- `chain_id`
- `authority=false`
- `truth_claim=false`

## Docket #57 Binding

Docket #57 is bound to ROS-0006.

The identity-binding gate SHALL be interpreted as:

> Identity binding MUST be proven against the Authorized Identity using the appropriate canonical verification path.

For Docket #57:

- Authorized Identity: `0xA380552a27b0a5a2874Ea7AA52CAC09f542002E8`
- Identity type: contract account
- Canonical path: ERC-1271
- Required magic value: `0x1626ba7e`

## Boundary

Signature verification proves identity binding only.

It does not create truth authority.

All compliant outputs MUST preserve:

```json
{
  "verdict": "WITNESS_ONLY",
  "authority": false,
  "truth_claim": false
}
```
